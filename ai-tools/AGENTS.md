# vinta-schedule-frontend-web

The web frontend for Vinta Schedule — a Next.js 16 App Router application (React 19,
TypeScript strict) that consumes an external Django REST API. It is a pure frontend: no
database, no server-side persistence of its own. All domain data comes from the backend
over generated OpenAPI clients.

Conventions for agents working in this repo. **UI/design rules live in `DESIGN.md` at the
repo root — read it before any visual work.** This file covers the non-visual conventions:
stack, commands, structure, code patterns, testing, and deploy.

## Project Overview

- **Root app** (`src/`) — the Next.js application: routes, feature compositions, data
  hooks, generated API clients.
- **`packages/design-system/`** — the `vinta-schedule-design-system` workspace package:
  shadcn/ui atoms (new-york style), layout primitives, design tokens, and its own
  Storybook. Consumed by the app via the package specifier, never by relative path.
- **`e2e/`** — Playwright end-to-end suite that drives the app against a live backend.

**There is no database in this repo.** Server state is the backend API's, fetched through
`src/client/` (app API) and `src/auth-client/` (allauth), both generated by hey-api and
cached by TanStack Query v5.

## Tech Stack

- **Next.js 16** (App Router, React Server Components) + **React 19** + **TypeScript**
  (strict).
- **Tailwind CSS v4** — tokens in `packages/design-system/src/styles/tokens.css`, no
  separate Tailwind config file.
- **Design system**: workspace package `vinta-schedule-design-system`
  (`packages/design-system/`) — shadcn/ui atoms, layout primitives, tokens. Icons:
  `lucide-react`.
- **TanStack Query v5** for server state.
- **hey-api** generated OpenAPI clients: `src/client/` (app API) and `src/auth-client/`
  (allauth). **Generated — never hand-edit.**
- **react-hook-form** + **zod** (`@hookform/resolvers/zod`) for forms.
- **Vitest** + **Testing Library** (jsdom) for unit tests.
- **Playwright** (`@playwright/test`) for e2e — chromium only, live backend, no mocks.
- Two **Storybooks**: app feature stories (`@storybook/nextjs-vite`, root `.storybook/`)
  and design-system stories (`@storybook/react-vite`,
  `packages/design-system/.storybook/`).
- **ESLint** (`eslint-config-next`) + **Prettier** (with `prettier-plugin-tailwindcss`).
- Package manager: **pnpm workspaces** (`pnpm-lock.yaml`, root app + `packages/*`), pinned
  via `packageManager` in `package.json`. Node version in `.nvmrc`.
- **Deploy target**: Vercel, driven by a version tag through GitHub Actions + the Vercel
  CLI. See the **Deployment** section.

## Code Style

- Path alias **`@/` → `src/`** (e.g. `@/lib/utils/index`). Design-system imports use the
  package specifier: `vinta-schedule-design-system/ui/button`,
  `vinta-schedule-design-system/layout`. Inside the design-system package, imports are
  relative.
- One hook per file, `use-kebab-name.ts`, named export `useCamelName`.
- All visual components get a colocated `*.stories.tsx`.
- Unit tests colocate as `*.test.ts(x)` next to the unit.
- Add short comments explaining the _why_ for non-obvious auth/cache behavior. Don't
  narrate the _what_.
- Prettier owns formatting (`pnpm run format:fix`). Don't hand-format.

## Design system

**`DESIGN.md` at the repo root is normative for all UI and visual work** — tokens, layout
primitives, responsive/container-query contracts, Tailwind v4 gotchas, icons, Storybook
conventions, and the pre-finish UI checklist. Read it before touching any component, page,
or story. It is not summarized here and must not be contradicted by this file; where the
two appear to disagree, `DESIGN.md` wins for anything visual.

primitives + props, never hand-rolled utility classes; `className` is an escape hatch

## Architecture

### Data hooks (`src/hooks/<domain>/`)

A hook is a thin wrapper over a generated TanStack Query operation. It spreads the
generated `*Mutation()` / `*Options()` factory, adds cache invalidation, and returns both
an ergonomic async fn and the raw mutation/query object.

```ts
import type { OrganizationWritable } from '@/client';
import { organizationsCreateMutation } from '@/client/@tanstack/react-query.gen';
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { CURRENT_ORGANIZATION_QUERY_KEY } from './use-current-organization';

export function useCreateOrganization() {
  const queryClient = useQueryClient();
  const createOrganizationMutation = useMutation({
    ...organizationsCreateMutation(),
    onSuccess: () => {
      queryClient.invalidateQueries({
        queryKey: CURRENT_ORGANIZATION_QUERY_KEY,
      });
    },
  });
  const createOrganization = async (body: OrganizationWritable) =>
    createOrganizationMutation.mutateAsync({ body });
  return { createOrganization, createOrganizationMutation };
}
```

- One hook per file, `use-kebab-name.ts`, named export `useCamelName`.
- Import types from `@/client` / `@/auth-client`; operation factories from the
  `@tanstack/react-query.gen` barrel.
- Export a `*_QUERY_KEY` const from query hooks so mutations can invalidate it.
- Add a short comment explaining the _why_ (non-obvious auth/cache behavior).

### Pages (`src/app/**/page.tsx`)

- **Server Component by default.** Fetch with the generated client directly in the async
  component (see `src/app/auth/login/page.tsx`). Degrade gracefully on API error — never
  redirect to the same route (infinite-loop risk).
- Add `'use client'` only when the file needs hooks/state/handlers.
- Keep pages thin: fetch + compose a feature component; put logic/markup in
  `src/components/<feature>/`.
- Dynamic segments: `[param]` folders; route handlers are `route.tsx`.

### Components

- **Layout primitives** (`packages/design-system/src/layout/`): token-prop driven, built on
  `Box`, `forwardRef`, re-exported from `index.ts`. No raw utility classes for layout —
  that's the whole point.
- **UI atoms** (`packages/design-system/src/ui/`): shadcn/ui, variants via `cva`, merge
  classes with `cn()`. Prefer `npx shadcn@latest add <name>` (run inside
  `packages/design-system/`) for stock components, then convert the generated `@/…` imports
  to relative ones.
- **Compositions** (`src/components/<feature>/`): assemble primitives + ui into a feature.
  `'use client'` when interactive; forms use react-hook-form + zod.
- All visual components get a colocated `*.stories.tsx`.

### Testing — unit

- Vitest + Testing Library, jsdom env, globals on (no imports of `describe`/`it` needed but
  existing tests import them explicitly — match the neighbor file).
- Colocate as `*.test.ts(x)` next to the unit. Mock `next/navigation` and network (`fetch`
  / generated client) as in `src/app/auth/social/finish-signup/page.test.tsx`.
- `vitest.setup.ts` clears `localStorage` + cookies between tests.

### Testing — e2e (Playwright)

Config: `playwright.config.ts`. Docs: `e2e/README.md`. Use-case index:
`QA_USE_CASES.md` at the repo root.

- **Tests run against a live backend. There is no mocking layer.** A spec without valid
  credentials and a reachable API fails by design — the shell's auth gate redirects away.
- `testDir` is `e2e/tests`; single **chromium** project (`Desktop Chrome`). `baseURL` comes
  from `E2E_BASE_URL`, defaulting to `http://localhost:3000`.
- `webServer` starts the dev server automatically, with `reuseExistingServer: true` — if
  you already have `pnpm run dev` up, Playwright reuses it.
- On CI (`process.env.CI`): `retries: 1`, `workers: 1`. Locally: no retries, default
  workers.
- Timeouts: 30s per test, 10s per expect. Trace on first retry, screenshot on failure,
  video off.
- Specs: `e2e/tests/app/PR000-smoke.spec.ts`, `PA001-team-list.spec.ts`,
  `PA002-invitations-list.spec.ts`. One spec per use-case; IDs are stable and never
  renumbered (`PR###` = member, `PA###` = admin).
- Helpers: `e2e/fixtures/auth.ts` (`memberPage` / `adminPage` fixtures — seed
  `localStorage.accessToken` with a real JWT), `e2e/page-objects/base-page.ts` (`BasePage`:
  `goto`, `screenshot`).
- Screenshots: specs call `basePage.screenshot('<id>', '<NN>', '<slug>')`; after a run,
  `node e2e/scripts/collect-screenshots.mjs` moves them into `pr-screenshots/`
  (gitignored).
- `e2e/tsconfig.json` is a separate TS config for the e2e tree.

Known inconsistency, documented not fixed: `playwright.config.ts` `webServer.command` is
`npm run dev` while the repo is pnpm-only. It works (npm can run the script) but is
off-convention. `e2e/README.md` also references a `.env.local.example` that does not exist
in the tree — the only committed example file is `.env.example`.

## Tenant model

Organization-scoped. Authorization and row-level isolation are the Django API's
responsibility — never reimplement or work around them client-side. But the frontend _does_
carry one load-bearing piece of the tenancy contract, and it has a sharp edge:

**`X-Organization-Id` is injected by a request interceptor, client-side only.**
[`src/lib/authentication-fetch-interceptors.ts`](../src/lib/authentication-fetch-interceptors.ts)
registers an interceptor that reads the active organization via `getActiveOrganizationId()`
(backed by `localStorage`) and sets `X-Organization-Id` on every outgoing request — unless the
header is already present, so an explicit per-request override wins.

- **It returns early when `typeof window === 'undefined'`, so the header is never injected on
  the server.** A fetch issued from a Server Component is therefore _not_ org-scoped by this
  path. If a server-side request must be org-scoped, pass the header explicitly — don't assume
  the interceptor covers you.
- Not every endpoint is org-scoped. `use-my-organizations.ts` documents its endpoint as
  deliberately un-scoped (it's the call that discovers which orgs you have).
- The active org is read from the API via `GET /organizations/current`, cached under
  `CURRENT_ORGANIZATION_QUERY_KEY`. `src/hooks/organizations/` owns this domain, including
  `use-organization-error-recovery.ts` for recovering from a stale/invalid selection.
- Reseller organizations add a whitelabel branding tier (`src/hooks/branding/use-branding.ts`).

This is a hot path — every API request goes through it. Changes here have wide blast radius;
`src/lib/authentication-fetch-interceptors.test.ts` covers the injection rules and is the first
thing to read before touching it.

## Environment Variables

**App** (`.env`, see `.env.example` — values are never committed):

```
NEXT_PUBLIC_API_BASE_URL
```

Base URL of the backend REST API. Used both at codegen time (`openapi-ts`) and at runtime
to point the generated API clients at the backend (see `src/lib/configure-api-clients.ts`).
Local dev defaults to `http://localhost:8000` when unset.

**E2E** (see the table in `e2e/README.md` for the authoritative list):

```
NEXT_PUBLIC_API_BASE_URL   # required — live/staging backend
E2E_MEMBER_ACCESS_TOKEN    # required — JWT for a member account in the QA org
E2E_ADMIN_ACCESS_TOKEN     # required — JWT for an admin account in the QA org
E2E_MEMBER_REFRESH_TOKEN   # optional
E2E_ADMIN_REFRESH_TOKEN    # optional
E2E_BASE_URL               # optional — defaults to http://localhost:3000
```

**Never commit tokens.** Rotate if compromised.

**CI secrets**: `VERCEL_TOKEN` (GitHub Actions secret, used by the production deploy).

## Dependency licenses

**Enforcement:** `block` — refuse install and ask the user before any override.

**Forbidden SPDX licenses** (any new third-party dep matching these must be checked before
install):

- `GPL-2.0-only`
- `GPL-3.0-only`
- `AGPL-3.0-only`
- `SSPL-1.0`

**Pre-install check.** Before running `pnpm add` (or `npm add` / `yarn add`) for any new
dependency, look up the package's declared license with `npm view <pkg> license` and
compare against the list above. If the license is in the list and the `(package, license)`
pair is not in **Approved overrides**, stop:

1. Do not run the install command.
2. Surface the violation to the user: package name, SPDX identifier, why it's forbidden,
   link to the upstream license.
3. Offer alternatives (search for an MIT / Apache-2.0 / BSD-licensed equivalent) before
   asking for an override.
4. If the user grants an override, record it in
   `policies.dependency_licenses.allowed_overrides[]` of `.vinta-ai-workflows.yaml`
   (package + SPDX + one-line reason) before re-running the install. Undocumented overrides
   leak into the diff and the reviewer will flag them.

**Unknown / undeclared license.** When the lookup returns no license, an empty value,
`UNKNOWN`, `SEE LICENSE IN <file>`, or only an unstructured `LICENSE` file with no SPDX
identifier, **stop and ask the user** regardless of enforcement mode. Don't guess, don't
fall back to "assume MIT" — the package may be all-rights-reserved by default. The user
picks one of: skip the dep and find a licensed alternative, treat as forbidden, or record
an `allowed_overrides` entry with an SPDX they independently confirmed off-channel.

Transitive dependencies follow the same policy, but checking every transitive license at
install time is impractical. The agent's responsibility is the **direct** add.

**Approved overrides:** none currently.

## Deployment

| Environment    | Target       | Trigger                             |
| -------------- | ------------ | ----------------------------------- |
| Production     | Vercel       | Push a version tag matching `v*`    |
| Storybook docs | GitHub Pages | Push to `main` (or manual dispatch) |

### Production (`.github/workflows/deploy-production.yml`)

**Vercel's Git integration is deliberately not used** — the repo lives under the
`vintasoftware` org while the Vercel project is on a different scope, so a Git-integration
deploy is not possible. Deploys run through the Vercel CLI with a `VERCEL_TOKEN` secret:

1. `vercel pull --yes --environment=production`
2. `vercel build --prod`
3. `vercel deploy --prebuilt --prod`

Triggered only by `push: tags: v*` — **branch pushes never deploy.** To ship, tag a commit
(`v1.2.3`) and push the tag. A `production-deploy` concurrency group with
`cancel-in-progress: false` prevents two tag deploys racing. `VERCEL_ORG_ID` /
`VERCEL_PROJECT_ID` are inline non-sensitive values in the workflow.

### Storybook (`.github/workflows/deploy-storybook.yml`)

Builds both Storybooks and publishes to GitHub Pages on push to `main`: app catalog at
`/<repo>/`, design system at `/<repo>/design-system/`. Base paths are injected via
`STORYBOOK_BASE_PATH`.

Known inconsistency, documented not fixed: the production workflow pins `node-version: 22`
while the storybook workflow pins `24`, and `.nvmrc` says `v22.16.0`.

## Git, commits, and PRs

- **Branch from `main`.** `main` is the default branch.
- **Commit style**: Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`).
- **PR creation**: agents create PRs (via `gh`) after review passes.
- Implementation plans and specs live in `ai-plans/`. There is no PR template in
  `.github/` — write a free-form Description.

## Hard rules

- **Never edit `src/client/` or `src/auth-client/`** — regenerate instead
  (`pnpm run openapi-ts` / `pnpm run openapi-ts-auth`).
- **No raw hex/rgb, no hand-rolled layout classes** — tokens + primitives (`DESIGN.md`).
- **Never `git add -A`** — the tree carries untracked `.env*` files, generated
  `schema.yml` / `schema-auth.yml` synced from the backend repo, and regenerated
  `src/client/` + `src/auth-client/` output. `git add -A` sweeps secrets and codegen churn
  into the commit. Stage explicit paths: `git add <explicit paths touched by this change>`.
- **No `Co-Authored-By` AI trailers** in commits.
- **Icon-only controls need `aria-label`.**
- **Never commit e2e tokens** or any real credential.
- **Check a new dependency's SPDX license before `pnpm add`** — see the **Dependency
  licenses** section.
